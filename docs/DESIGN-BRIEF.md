# Rhapsody 设计纲领

> 文档状态：Approved 1.0（设计纲领定稿）<br>
> 文档级别：最高设计约束<br>
> 项目：Rhapsody · 狂想曲

这份文档定义 Rhapsody 的创作意图和审美边界。未来的页面设计、组件选择、动效、技术实现与素材引用，都应当能够从这里获得理由。当局部效果与本纲领冲突时，以本纲领为准。

---

## 1. 项目命题

Rhapsody 不是一组堆叠在 SillyTavern 上的特效，也不是对某个 Awwwards 网站的复刻。

它希望回答的是：

> 如果聊天界面也能像一首结构完整的乐曲、一首含蓄的诗那样被感知，它应当如何呼吸、停顿、展开和收束？

Rhapsody 将 SillyTavern 原生界面视为已经存在的乐器。扩展不拆毁乐器，也不改写其演奏机制，而是重新组织舞台、光线、音色与节奏，使用户获得更完整的视觉体验。

## 2. 名称与精神

### 2.1 Rhapsody · 狂想曲

“Rhapsody”既包含自由、想象与情绪流动，也保留音乐所要求的内部秩序。

这里的“狂想”不等于无节制。真正的自由来自对比例、节奏、重复、变化和停顿的控制。Rhapsody 可以拥有强烈的视觉时刻，但这些时刻必须被安静的界面包围，正如乐曲中的高潮需要前后的铺垫与余韵。

它也是创作者自己的狂想曲：个人审美可以鲜明存在，却不以压过内容和用户为代价。

### 2.2 两个主题，两个乐章

Rhapsody 由两个主题构成：

#### Nocturne · 夜曲

Nocturne 不是纯黑、霓虹和赛博光效的组合。它应当表现夜晚的深度、内省和微弱光线：

- 深色中仍然存在可辨认的层次；
- 光来自局部，而不是所有边缘同时发亮；
- 动作缓慢、低声、具有余韵；
- 色彩倾向冷紫、夜蓝和接近月光的灰白；
- 高亮只出现在真正重要的状态与操作上。

Nocturne 的关键词是：**沉静、纵深、回声、月光、内省**。

#### Aubade · 晨曲

Aubade 不是简单地把 Nocturne 反相为白色。它应当表现黎明的空气、温度与逐渐清晰的边界：

- 基底为温暖的象牙白、雾灰和极浅的蓝；
- 光线柔和扩散，不使用刺眼的纯白大面积照明；
- 层次依靠明度、细线和轻微阴影建立；
- 动作比 Nocturne 更轻、更清醒，但仍然克制；
- 少量暖金可以提示日光，却不能演变为奢华装饰。

Aubade 的关键词是：**黎明、空气、苏醒、清澈、温度**。

### 2.3 共同的主题基因

Nocturne 与 Aubade 必须共享：

- 同一套信息层级；
- 同一套排版比例；
- 同一套间距节奏；
- 同一套组件结构与交互逻辑；
- 同一种克制、含蓄、重视内容的态度。

主题切换改变的是光线与情绪，不是产品性格。

## 3. 核心美学原则

### 3.1 和谐不是平均

和谐来自不同元素之间有清楚的关系，而不是让所有元素拥有相同的视觉重量。

一个页面应当有主声部、伴奏和休止：

- 当前内容是主声部；
- 导航与工具是伴奏；
- 环境光、纹理和装饰是几乎不可察觉的背景；
- 留白是休止，而不是尚未填满的空位。

### 3.2 均衡不是机械对称

Rhapsody 允许不对称构图，但视觉重心必须稳定。大块留白可以由小面积高亮平衡；沉重的深色面板可以由轻薄的线条与空气感缓解。

任何结构调整都应先解决阅读与操作，再解决画面构图。

### 3.3 节奏来自重复与变化

间距、圆角、标题比例、消息之间的停顿、按钮反馈的速度，都应形成可感知的节奏。

重复建立秩序，变化建立情绪。没有秩序的变化只是噪声，没有变化的重复则会僵化。

### 3.4 克制产生高级感

高级感不来自更多的模糊、更亮的边缘或更多 WebGL 实例，而来自知道何时停止。

同一视口内，应尽量只有一个主要动态焦点。背景、角色立绘、边框和文字不应同时争夺注意力。

### 3.5 内容始终先于装饰

用户来到 SillyTavern 是为了角色、文字与对话。Rhapsody 的视觉表现应当帮助用户进入情境，而不是让用户持续意识到插件正在表演。

当内容密度提高时，装饰必须自动后退。

## 4. 诗歌性的表达方式

Rhapsody 对诗歌的参考不是在界面上大量使用诗句，而是采用诗歌组织意义的方式：

- 用暗示代替解释；
- 用停顿保留余地；
- 用意象连接情绪，而不是说明功能；
- 允许局部不完全显露；
- 让同一个视觉母题在不同页面中产生轻微变奏。

界面文案必须清楚。诗意存在于整体感受和少量非功能性文字中，不能妨碍用户理解按钮、状态与错误信息。

## 5. 神话元素的距离

神话是 Rhapsody 的远景，不是舞台前景。

它可以像一段难以辨认的旧故事、一组星辰之间可能存在的连线，或某种被时间磨淡的象征。用户不需要知道它来自哪一位神祇、哪一个传说，也不需要阅读世界观才能理解界面。

### 5.1 可以使用

- 极淡的星图、轨迹和天体弧线；
- 接近铭文比例的细线与几何秩序；
- 光环、月相、晨曦等跨文化的自然意象；
- 隐约呼应桂冠、羽翼、海浪或织线的抽象曲线；
- 少量具有古典韵律的命名；
- 只有在静止观察时才可能发现的背景细节。

### 5.2 不应使用

- 大面积希腊柱、神殿、雕像或壁画；
- 把具体神祇作为主要角色或品牌标志；
- 金色纹章、繁复花边和仿古装饰的堆叠；
- 为解释隐喻而增加大段世界观文案；
- 让神话符号比聊天内容和角色本身更醒目；
- 将“古典”误解为奢华、陈旧或博物馆式复古。

### 5.3 判断标准

如果用户第一眼就能明确说出某个神话元素是什么，它通常已经太近、太重。理想状态是先感受到一种遥远的历史与想象，之后才可能发现构成这种感受的细节。

## 6. 视觉语言

### 6.1 排版

排版是 Rhapsody 最重要的视觉材料之一。

- 功能信息优先保证清晰、稳定和长时间阅读舒适；
- 展示性字体只用于少量标题和过渡画面；
- 中文与西文必须分别校准字重、行高和字间距；
- 不通过无限放大标题制造所谓高级感；
- 字号与留白应形成类似乐句的层级和停顿。

### 6.2 色彩

- 避免未经调和的纯黑与纯白；
- 每个主题设置明确的背景、表面、正文、次级文字、强调和状态色；
- 强调色是稀缺资源，不用于普通边框和所有可交互元素；
- 色彩必须满足可读性要求，氛围不能以牺牲文字对比度为代价。

### 6.3 材质

玻璃、模糊、噪点、光晕和扫描纹理只能作为层级工具：

- 模糊用于区分前景与环境；
- 噪点用于削弱过于数字化的平滑感；
- 光晕用于提示焦点与空间深度；
- 边框用于定义结构，而不是持续发光；
- 材质必须在低性能模式下能够安全移除。

### 6.4 图像与图标

- 核心界面不依赖远程图片；
- 图标优先使用本地 SVG；
- 简单纹理优先由 CSS 或程序生成；
- 用户自定义背景图片属于可选能力，不构成主题身份；
- 任何角色图像效果都必须保留静态回退。

## 7. 动效如同速度与力度

动画应当拥有音乐中的速度、力度和句法，而不是平均地应用于所有元素。

### 7.1 动效层级

1. **功能反馈**：按钮、输入、展开、关闭。速度最快，必须明确。
2. **结构过渡**：抽屉、设置中心、角色切换。速度适中，帮助理解空间变化。
3. **叙事时刻**：首次加载、空白欢迎区。可以更有表现力，但必须短暂且可跳过。
4. **环境运动**：粒子、光线、Flowmap。最慢、最弱，不承担关键反馈。

### 7.2 动效纪律

- 动画不能延迟用户已经可以执行的操作；
- 加载动画绑定真实初始化状态，不制造固定时长的假等待；
- 同一时间避免多个高强度循环动画；
- 页面隐藏、效果离屏或组件卸载后必须停止绘制；
- 提供完整、减弱和关闭动态效果的能力；
- 遵循 `prefers-reduced-motion`。

## 8. SillyTavern 的结构边界

Rhapsody 是视觉扩展，不是 SillyTavern Fork。

### 8.1 保持不变

- 数据结构与持久化机制；
- API 与事件语义；
- 消息生成、编辑和流式输出逻辑；
- 角色、聊天和设置的核心行为；
- 关键 DOM 节点的身份与父子关系；
- 其他扩展依赖的公开挂载点。

### 8.2 允许的视觉增强

- 通过作用域 CSS 与设计令牌重构色彩、间距、排版和材质；
- 使用 Grid、Flex 与 `order` 在不搬动节点的前提下调整视觉布局；
- 为插件自己的设置中心、欢迎区和加载层创建独立挂载节点；
- 通过只读状态监听同步主题与界面状态；
- 使用 `MutationObserver` 处理动态出现的原生元素；
- 在禁用插件时完整移除 class、属性、事件、观察器与 Canvas。

### 8.3 谨慎或禁止

- 大规模使用 `appendChild()` 重新排列核心节点；
- 克隆消息或输入区域并隐藏原始组件；
- 覆盖酒馆业务函数以适配视觉效果；
- 让视觉顺序与键盘焦点顺序严重不一致；
- 依赖未经记录的全局变量和短期 DOM 偶然结构。

## 9. 效果的角色分配

效果库是调色板，不是页面清单。任何效果进入正式界面前，都必须回答“它在此处承担什么职责”。

- 程序化背景负责环境，不承担内容；
- 角色立绘交互负责角色进入与切换，不用于消息列表头像；
- 碎片聚合只用于插件初始化或少量叙事过渡；
- 发光边框只提示焦点和重要状态；
- 展示型菜单只用于独立主题中心，不替换高频原生导航；
- 同一页面优先选择一个主要效果，而不是叠加所有素材。

## 10. 体验与性能底线

无论视觉目标多高，以下条件不可被牺牲：

- 聊天文字始终清晰可读；
- 输入、发送、编辑和滚动不因动效而变慢；
- 移动端能够完成全部核心操作；
- 键盘焦点可见，状态不只依赖颜色表达；
- 动效可以降低或关闭；
- 核心视觉可以完全离线运行；
- 单个效果失败时，酒馆原生功能仍然可用；
- 插件关闭后，界面能够恢复到启用前状态。

## 11. 设计评审问题

每个页面、组件和动效在进入实现前，应依次回答：

1. 它首先帮助用户完成什么？
2. 视觉主次是否清楚？
3. 它属于主声部、伴奏、休止还是背景？
4. Nocturne 与 Aubade 是否共享同一结构语言？
5. 神话隐喻是否足够遥远？
6. 是否存在为了展示技术而加入的效果？
7. 去掉这个效果后，信息是否仍然完整？
8. 减少动态效果时，体验是否仍然成立？
9. 是否保持了 SillyTavern 的原生结构和交互？
10. 用户长时间聊天时，它是否仍然舒适？

任何无法回答的问题，都不应通过增加更多装饰来掩盖。

## 12. 后续文档顺序

在本纲领确认后，按以下顺序推进：

1. **视觉基底与设计语言**：把审美原则转化为编辑器骨架、双主题性格和加载界面方向。
2. **双主题设计令牌**：颜色、字体、空间、圆角、阴影、材质、动效速度。
3. **页面骨架**：主聊天界面、角色欢迎页、插件设置中心、初始化覆盖层。
4. **交互规范**：主题切换、角色切换、抽屉与弹窗、加载与退出。
5. **适配规范**：SillyTavern 选择器、动态节点、兼容策略与清理机制。
6. **性能与无障碍规范**：档位、回退、移动端、键盘与减少动态效果。
7. **实施计划**：按风险从纯 CSS 到独立 React 表面逐步落地。

## 一句话准则

> Rhapsody 应当像一首被精心控制的狂想曲：情绪自由，结构克制；光与运动可以被感受到，却永远不盖过角色、文字与对话。
